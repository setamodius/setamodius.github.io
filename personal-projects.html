Personal Projects

Smaller in scope, but where most of the actual learning happens — no deadline, no committee, just a problem worth solving.

Pharmatherm

Motivation

Maintaining the cold chain in pharmacies is critical — a few hours of refrigerator failure can ruin temperature-sensitive medication. Pharmatherm grew out of that real, specific need: a device that watches refrigerator temperature continuously and tells someone the moment things go wrong, without relying on a person checking a thermometer.

Architecture

Custom hardware reads temperature sensors over Bluetooth and forwards readings to a cloud-based server. If the temperature crosses a configured threshold, the system sends an SMS alert to the people responsible. Because internet connectivity at a pharmacy can't be assumed to be reliable, readings are also written to an SD card locally — so a connection drop doesn't mean a data gap, and the buffered data syncs to the server once the connection is restored.

Lessons learned

The SD-card buffering turned out to be the most important design decision in the whole project — it's a small reminder that "the cloud" is not a substitute for handling the case where the network just isn't there.

Janus Lock

Motivation

Janus Lock was created to simplify access to my garage while preserving the security mechanisms already provided by the original rolling-code remote control. Rather than attempting to replicate or reverse engineer the proprietary radio protocol, the goal was to integrate modern connectivity on top of the existing access system in a reliable, non-invasive way.

Architecture

The system is built around an ESP32-C3 microcontroller with an integrated OLED display and MQTT connectivity. A modified rolling-code remote control remains responsible for all RF communication with the garage door — Janus Lock electronically actuates the original remote's push button using a CPC1017N solid-state relay, effectively emulating a button press while preserving the manufacturer's security model. Configuration is handled through a lightweight web interface, with operational status shown on the onboard OLED screen.

Lessons learned

This project reinforced an important engineering principle: sometimes the most reliable solution isn't replacing an existing system, but integrating with it. By leveraging the original remote rather than reimplementing its protocol, the solution stayed secure, maintainable, and resilient while gaining modern IoT capabilities.

BLE Door Access System

Motivation

This project automates access to the entrance door of my apartment building without requiring a smartphone application, cloud service, or manual interaction. The objective was a simple, reliable, proximity-based access system that operates entirely locally.

Architecture

An ESP32 microcontroller continuously scans for an authorized BLE beacon and evaluates signal strength (RSSI) to estimate proximity. When the beacon enters a predefined range, the system triggers the door release mechanism through a relay output. The software includes connection monitoring, signal filtering, reconnection logic, and status feedback through LEDs and audible indicators.

Lessons learned

The project provided valuable experience in BLE communication, signal-strength-based decision making, and embedded event-driven software design. One of the most important observations was the challenge of translating unreliable radio signals into reliable real-world actions — achieving consistent behavior required careful filtering, timing logic, and fault tolerance mechanisms.

Home Automation

Motivation

Two small, separate annoyances that turned out to share a solution: watering plants on a schedule that actually matched what they needed, and a house that didn't adjust its own lighting or heating without being told to.

Architecture

An automated irrigation controller reads soil moisture and waters on a programmable schedule via an LCD interface, rather than on a fixed timer that ignores actual conditions. Separately, a lighting and heating system built on ESP32 and Sonoff devices adjusts home lighting based on sunset time and controls the boiler based on indoor temperature — a simple, low-cost alternative to commercial home automation hubs.

Lessons learned

Sensor-driven automation beats fixed schedules almost every time, but only if the sensor itself is trustworthy — most of the iteration on both projects went into making the moisture and temperature readings stable rather than into the control logic itself.

ESP32 & MQTT Experiments

Motivation

A standing playground for testing ideas that are too small or too exploratory for a "real" project — a new sensor, a different MQTT broker configuration, a power-management trick — before they get folded into something like Janus Lock, the BLE door access system, or the home automation setup.

Architecture

Mostly ESP32-based nodes publishing to a local MQTT broker, used to test publish-subscribe patterns, payload formats, QoS behavior, and low-power sleep cycles outside the constraints of a production system.

Lessons learned

This is where most of the practical MQTT and embedded knowledge that ends up in professional SCADA work actually gets tested first — it's cheaper to learn that a sleep mode breaks reconnection logic on a $5 board on a desk than on a deployed system.
